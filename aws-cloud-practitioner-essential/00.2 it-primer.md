# IT Primer for the CLF-C02 (plain-English fundamentals)

> The exam assumes you know basic IT vocabulary. This file is for the bits the AWS course skips. Re-read once before exam day.

## 1. Network layers (only 3 matter)

A network connection is like sending a letter through layers of inspection:

| Layer | Name | What lives here | Letter analogy | What firewalls at this layer check |
|---|---|---|---|---|
| **3** | Network | IP addresses, routing | Street address on the envelope | "Is this from a blocked address/country?" |
| **4** | Transport | TCP, UDP, **ports** (80, 443, 22, 3306, …) | Department/floor at that address | "Is this destined for an open port?" |
| **7** | Application | HTTP/HTTPS, the actual request body | The letter inside the envelope | "Does this letter contain SQL injection? XSS? a bad URL?" |

Layers 1, 2, 5, 6 exist (cables, MAC addresses, sessions, encoding) but **don't appear on the exam**. Ignore them.

Rule of thumb: the **deeper the layer (3 → 7)**, the smarter and slower the inspection. Layer 3 just looks at the envelope. Layer 7 opens the letter and reads it.

### Common ports you should recognize

| Port | Protocol |
|---|---|
| 22 | SSH (Linux remote login) |
| 3389 | RDP (Windows remote login) |
| 80 | HTTP |
| 443 | HTTPS (HTTP + TLS) |
| 3306 | MySQL / MariaDB / Aurora MySQL |
| 5432 | PostgreSQL / Aurora Postgres |
| 1433 | SQL Server |
| 1521 | Oracle |

If a question says "open port 22 to the world" — that's exposing SSH to the internet, almost always wrong. "Open port 443" — normal HTTPS web traffic, fine.

## 2. AWS firewalls (the office building analogy)

Imagine your AWS account is an office building. Each firewall is a different checkpoint:

| Firewall | Where it sits | Layer | Building analogy |
|---|---|---|---|
| **Shield** | At the building's perimeter | 3/4 (Standard); 3/4 + 7 (Advanced) | Riot police outside — stops a **mob** (DDoS) before it reaches the door |
| **WAF** | At the front door | **7** | Bouncer who opens every package and reads the contents (blocks SQLi, XSS) |
| **Network ACL (NACL)** | At each floor's elevator lobby | 3/4 | Security desk on each **floor (subnet)** — checks IDs in and out |
| **Security Group** | At each office door | 3/4 | Lock on each **office (EC2 instance)** — only approved visitors |
| **Network Firewall** | The building's central mail room | 3-7 | Inspects all traffic entering/leaving the whole **building (VPC)** |
| **Firewall Manager** | Corporate security HQ | — | Central console that pushes the same rules to every **building (account)** you own |

### Security Group vs NACL (both sit inside your VPC)

| | Security Group | NACL |
|---|---|---|
| Protects | One **EC2 instance** (or ENI) | A whole **subnet** |
| Stateful? | **Stateful** — reply traffic auto-allowed | **Stateless** — must allow both directions |
| Rule types | **Allow only** | **Allow and deny** |
| Default behavior | Deny all in, allow all out | Default NACL = allow all; custom NACL = deny all |

Memory: **S**ecurity Group = **S**ingle instance, **S**tateful. **N**ACL = **N**etwork (subnet), **N**ot stateful.

### WAF vs Shield (both sit in front of your app)

| | WAF | Shield |
|---|---|---|
| Layer | **7** (application) | **3/4** (Standard); 3/4 + 7 (Advanced) |
| Stops | App-level attacks: SQLi, XSS, bad bots, bad IPs, rate limits | **Volume-based DDoS** floods |
| Cost | Pay per rule + request | Standard = **free** for all customers; Advanced ≈ $3000/mo |
| Bonus on paid tier | — | Advanced: 24/7 DDoS Response Team, **cost protection** (refunds on traffic spikes from attacks), advanced reporting |

Memory: **W**AF = **W**ords in the request (Layer 7). **Shield** = volume of requests.

### Network Firewall vs WAF

- **Network Firewall** = at the VPC level, inspects **all** traffic (HTTP, DNS, anything) flowing in/out of your VPC. Think of it as a stateful IPS.
- **WAF** = only sits in front of **HTTP-based** services (CloudFront, ALB, API Gateway, AppSync). Doesn't see non-HTTP traffic.

If a question says "deep packet inspection across all VPC traffic" → Network Firewall. If it says "block SQL injection on my web app" → WAF.

## 3. Decision tree for any firewall question

Ask: **"What kind of attack or check?"**

- Block **SQL injection / XSS / bad URLs / bad bots** → **WAF**
- **DDoS** (volumetric flood from many IPs) → **Shield** (Standard for free auto, Advanced if they mention 24/7 team, cost protection, or layer-7 DDoS)
- Restrict **which ports/IPs can reach this EC2** → **Security Group**
- Restrict traffic at the **subnet** level / I need a **deny** rule / I need to block one bad IP without touching everything → **NACL**
- **Deep packet inspection across the whole VPC** / IPS for non-HTTP traffic → **Network Firewall**
- Apply the **same WAF/SG/Shield rules across many accounts** → **Firewall Manager**
- Just see SSL/TLS certs for free → **Certificate Manager (ACM)** (not a firewall, but appears in the same questions)

## 4. Storage, from your laptop up to AWS

### 4a. Your laptop, decoded

Your laptop has an SSD (or HDD). That's **block storage** — raw storage divided into fixed-size blocks. You **format** it once, which writes a **filesystem** on top (NTFS on Windows, ext4 on Linux, APFS on Mac). The filesystem is what turns "blocks" into the familiar files-and-folders layout.

**The OS itself is just a big pile of files** sitting in special folders on that same disk (`C:\Windows\` on Windows, `/usr/` and `/etc/` on Linux). When you press the power button:

1. Motherboard firmware (UEFI / BIOS) wakes up
2. It reads a small **bootloader** from a known location on the disk
3. The bootloader loads the OS kernel → OS takes over

So "where is the OS saved?" → **on the same block storage as everything else**, in folders the bootloader knows where to find.

### 4b. EC2 is just a virtual laptop

An EC2 instance is a VM running on AWS-managed hardware. It needs a "disk" to boot from, and AWS gives it one:

```
┌─────────────────────────────────────────┐
│ EC2 instance (a VM)                     │
│                                         │
│  Root EBS volume  ← boot disk, made     │
│   • OS files       from the AMI         │
│   • pre-installed                       │
│     software                            │
│                                         │
│  Extra EBS volume(s) (optional)         │  ← like adding a 2nd SSD
│  Instance Store (optional, EPHEMERAL)   │  ← scratch space, lost on stop
└─────────────────────────────────────────┘
        │
        ├──→ EFS  (shared NFS, mounted by many Linux EC2s)
        └──→ S3   (HTTPS API, not mounted as a drive)
```

### 4c. What is an AMI exactly?

An **AMI (Amazon Machine Image)** is **not a live disk** — it's a **frozen template**. Specifically, an AMI = a snapshot of a fully-configured root volume + metadata about how to launch it.

Analogy:

| Your laptop | AWS |
|---|---|
| Windows install ISO + a backup image of a perfectly-configured PC | **AMI** |
| The actual SSD inside the running laptop | **EBS root volume** |
| The laptop itself | **EC2 instance** |

When you launch an EC2 from an AMI, AWS:

1. Reads the AMI's snapshot
2. **Creates a fresh EBS volume** by copying the snapshot
3. Attaches it to your new EC2 as the **root (boot) disk**
4. Powers on the EC2 → OS boots from that volume

You can launch 100 EC2s from the same AMI → you get **100 independent EBS volumes**. Each starts identical, then evolves separately. You can also **make your own AMI** from a running EC2 ("Create Image") to spin up identical copies later — this is how Auto Scaling launches new instances.

### 4d. The four storage flavours, mapped to your laptop

| You want… | Laptop equivalent | AWS service | Persists if EC2 stops? |
|---|---|---|---|
| Boot disk for **this** EC2 | The C: drive inside your laptop | **EBS** (root volume) | Yes |
| Extra disk for **this** EC2 (database, app data) | A second internal SSD | **EBS** (extra volume) | Yes |
| Super-fast scratch space, OK if it disappears | RAM disk / temp drive | **Instance Store** | **No — gone on stop** |
| Folder that **many Linux** servers read+write at once | NAS at home that 4 laptops all mount | **EFS** | Yes |
| Same idea but Windows / SMB | A Windows file share | **FSx for Windows** | Yes |
| Same idea but high-performance computing | A Lustre cluster | **FSx for Lustre** | Yes |
| Dump unlimited files; access via API/HTTPS | **Dropbox / Google Drive** | **S3** | N/A — independent of EC2 |
| Cheapest archive for 7 years, retrieval in hours | A box of DVDs in your basement | **S3 Glacier Deep Archive** | N/A |

### 4e. The three storage **categories** (the AWS exam uses these terms)

| Category | What it is | Mountable like a drive? | AWS service |
|---|---|---|---|
| **Block** | Raw disk; the OS puts a filesystem on it | Yes, by **one** server | **EBS**, Instance Store |
| **File** | Shared network drive; many servers mount it | Yes, by **many** servers | **EFS** (NFS, Linux), **FSx** (SMB / Lustre / NetApp / ZFS) |
| **Object** | Upload/download whole files via API | **No** — it's HTTP, not a disk | **S3** |

### 4f. Exam shortcuts

When you see these phrases, the answer is automatic:

- "boot the EC2" / "OS disk" → **EBS** root volume
- "one server needs more disk space" → **EBS** extra volume
- "**fastest possible** disk, OK if data disappears" → **Instance Store**
- "**multiple Linux** EC2s read/write the same files" → **EFS**
- "Windows file share / Active Directory integrated" → **FSx for Windows**
- "high-performance computing / ML training shared filesystem" → **FSx for Lustre**
- "store **billions of objects**, accessed via HTTPS, infinite scale" → **S3**
- "static website hosting" → **S3** (yes, S3 can host websites)
- "data lake on a budget for analytics" → **S3** (with Athena / Glue on top)
- "archive for 7 years, retrieval in 12 hours OK, cheapest" → **S3 Glacier Deep Archive**
- "back up many AWS services from one place" → **AWS Backup**
- "on-prem appliance backed by S3" → **Storage Gateway** (File / Volume / Tape)
- "**physically** ship petabytes to AWS" → **Snowball Edge** (or Snowmobile for exabytes)

## 5. The compute family (what do you give AWS, and what do they manage?)

Compute services are confusing because they overlap. The cleanest way to think about them: **what do you hand to AWS, and how much do they manage for you?** The more you hand over, the less you control.

| You hand AWS… | Service | What AWS manages | Laptop analogy | When to use |
|---|---|---|---|---|
| Money + a use case | **Lightsail** | Everything: OS, server, networking, simple console | Renting a pre-configured laptop with flat monthly bill | Personal blog, small WordPress site, dev/test |
| An **OS image** (AMI) | **EC2** | The host hardware. **You manage the OS, patches, scaling, everything inside the VM.** | Renting a blank laptop and installing your own OS | Total control; lift-and-shift from on-prem; legacy apps |
| **Source code** | **Elastic Beanstalk** | OS, web server, load balancer, auto-scaling — picks a default stack for you | Handing your code to a sysadmin who builds the server for you | Web apps when you don't want to learn AWS plumbing |
| A **container image** (Docker) | **ECS** (AWS's orchestrator) **or** **EKS** (Kubernetes) | Cluster scheduling, networking, scaling — but **you still pick whether containers run on EC2 or Fargate** | Handing pre-packaged apps to someone who runs them in standard boxes | Microservices, Docker workloads |
| A container image, want **zero servers** | **Fargate** (used inside ECS or EKS) | Everything below the container — no EC2 to patch | Same as above but you don't even see the boxes | Containers without server management |
| A **function** (200ms of code) | **Lambda** | Everything. You don't even see a "server." | Submitting a single recipe, AWS cooks it on demand | Event-driven, glue code, APIs, automation |
| A **batch job script** | **AWS Batch** | Job queues + dynamic compute provisioning | "Run these 10,000 jobs whenever capacity is free" | HPC, scientific computing, video transcoding |

### 5a. The control vs management spectrum

```
MORE control                                                    LESS management
   |                                                                       |
   v                                                                       v

  EC2  →  ECS on EC2  →  ECS on Fargate  →  Beanstalk  →  Lambda  →  Lightsail / SaaS

  (raw VM)   (containers,   (containers,    (PaaS,         (FaaS,    (turnkey)
              you own EC2)   serverless)     code only)     functions)
```

### 5b. Lambda specifics that show up on the exam

- **Pay per millisecond** of execution + per request. No idle cost.
- **Max 15 minute** runtime per invocation.
- **Triggered by events**: S3 upload, DynamoDB stream change, API Gateway request, EventBridge schedule (cron), SNS/SQS message, etc.
- Languages: Node.js, Python, Java, .NET, Go, Ruby, custom runtimes.
- Common pattern: **API Gateway → Lambda → DynamoDB** = a complete serverless API.

### 5c. ECS vs EKS vs Fargate (the container confusion)

- **ECS** = AWS's own container orchestrator. Simpler, AWS-specific, less to learn.
- **EKS** = managed **Kubernetes**. Use this if your team already knows K8s or wants portability across clouds.
- **Fargate** = a "compute mode" for ECS or EKS that means "I don't want to manage the underlying EC2 hosts." Both ECS and EKS can run on either **EC2 launch type** (you manage hosts) or **Fargate launch type** (serverless containers).
- **ECR** = the container **image registry** (where you push your Docker images). Like Docker Hub but private.

So a typical question pattern: "We want to run containers without managing servers" → **Fargate** (with ECS or EKS).

### 5d. The "hardware in unusual places" trio

These three look similar but each solves a different problem:

| Service | What it is | When |
|---|---|---|
| **Outposts** | AWS hardware racks installed **in your own data center** | Regulatory / latency requires gear physically on-prem |
| **Local Zones** | AWS infrastructure in **specific metros** (e.g. Las Vegas, Boston) closer to users than the parent Region | You need single-digit ms latency in a city without a full Region |
| **Wavelength** | AWS compute embedded **inside 5G telco networks** (Verizon, KDDI, etc.) | Mobile/edge apps that need ultra-low latency over 5G |

### 5e. Compute exam shortcuts

- "Run code on a schedule, no servers" → **Lambda** (triggered by EventBridge cron)
- "Fast lift-and-shift VM, total control of OS" → **EC2**
- "Just upload my Python/Node code, AWS picks the stack" → **Elastic Beanstalk**
- "Run containers, don't want to manage EC2 hosts" → **Fargate**
- "Already know Kubernetes, want it managed" → **EKS**
- "AWS-native container orchestration, simplest" → **ECS**
- "Tiny WordPress / blog, flat monthly fee, simple console" → **Lightsail**
- "Massive batch jobs, queue them up" → **AWS Batch**
- "AWS hardware in **my** data center" → **Outposts**
- "Compute inside a 5G network" → **Wavelength**

## 6. The database family (matching your data shape to a service)

The trick with AWS databases: **the data shape decides the service**. Pick the right shape first, then pick the engine.

### 6a. Three shapes of data

| Shape | What it looks like | Tradeoffs | AWS service category |
|---|---|---|---|
| **Relational (SQL)** | Rows + columns + multiple tables linked by keys, joins, transactions | Strong consistency, mature, slower at huge scale | **RDS**, **Aurora**, **Redshift** |
| **Key-value / Document (NoSQL)** | One key → one blob (JSON, item attributes). No joins. | Massive horizontal scale, very fast lookups, less rich queries | **DynamoDB**, **DocumentDB**, **Keyspaces** |
| **Specialized** | Graph (nodes + edges), time-series, ledger, in-memory cache | Optimized for one job, bad at others | **Neptune**, **Timestream**, **QLDB**, **ElastiCache** |

### 6b. The relational family (SQL)

| Service | What | When |
|---|---|---|
| **RDS** | Managed standard relational engines: **MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, Aurora**. AWS handles backups, patching, replicas, failover. | You want a normal SQL database without running it yourself. |
| **Aurora** | AWS's own MySQL/PostgreSQL-**compatible** engine. Cloud-native architecture, 5× faster than MySQL, storage auto-scales to 128 TB, 6 copies across 3 AZs by default. | Production SQL workloads on AWS where performance / availability matter. |
| **Aurora Serverless** | Aurora that automatically scales capacity up/down, even to zero. Pay per usage. | Spiky or unpredictable workloads, dev/test, infrequent apps. |
| **Redshift** | **Data warehouse**, columnar, petabyte-scale, **OLAP** (analytics). SQL on huge historical data. | Reports / BI / analytics over millions to billions of rows. **Not** for live app traffic. |

**OLTP vs OLAP** (a phrase that shows up):
- **OLTP** = Online Transaction Processing → live app: orders, users, inventory. **RDS / Aurora.**
- **OLAP** = Online Analytical Processing → reporting and analytics. **Redshift.**

### 6c. The NoSQL / non-relational family

| Service | Data shape | Killer feature | When |
|---|---|---|---|
| **DynamoDB** | Key-value + document (JSON) | Serverless, single-digit ms at any scale, no servers to size | Carts, sessions, user profiles, IoT state, anything that needs huge scale + simple lookups |
| **DocumentDB** | JSON documents | **MongoDB-compatible** | You're already using MongoDB |
| **Keyspaces** | Wide-column | **Cassandra-compatible** | You're already using Cassandra |
| **Neptune** | Graph (nodes + edges) | Stores and queries relationships natively | Social networks, fraud detection, knowledge graphs, recommendation engines |
| **QLDB** | Ledger | Immutable, cryptographically verifiable history | Audit trails, financial records, supply-chain provenance |
| **Timestream** | Time-series | Optimized for time-stamped data | IoT sensor data, metrics, DevOps monitoring |

### 6d. The cache layer (in-memory, very fast)

| Service | What | When |
|---|---|---|
| **ElastiCache for Redis** | Managed Redis — rich data structures, pub/sub, persistence option | Speed up reads on a slow DB, leaderboards, real-time analytics, session store |
| **ElastiCache for Memcached** | Managed Memcached — simpler, multi-threaded, no persistence | Pure read-cache for a slow DB |
| **MemoryDB for Redis** | Redis-compatible but **durable** (survives reboots) | When you want Redis as a primary database, not just a cache |
| **DAX (DynamoDB Accelerator)** | In-memory cache **specifically for DynamoDB** | Push DynamoDB read latency from milliseconds → microseconds |

Common confusion: **ElastiCache** is a general cache for any backend. **DAX** is a cache **only** for DynamoDB.

### 6e. The "X vs Y" exam pairs

| Pair | The distinction |
|---|---|
| **Aurora vs RDS** | Aurora is AWS-built, faster, more available, slightly more expensive. RDS is the broader managed-SQL umbrella (which includes Aurora as one engine option). |
| **Redshift vs RDS** | Redshift = analytics warehouse (OLAP). RDS = transactional app DB (OLTP). |
| **DynamoDB vs RDS** | DynamoDB = NoSQL key-value, infinite scale, no joins. RDS = SQL relational, joins/transactions, vertical scale. |
| **DynamoDB vs DocumentDB** | DynamoDB = AWS-native key-value. DocumentDB = MongoDB-compatible document store (use only if you already speak Mongo). |
| **ElastiCache vs DAX** | ElastiCache = general cache. DAX = DynamoDB-specific cache. |
| **DynamoDB vs MemoryDB** | DynamoDB = persistent NoSQL on disk. MemoryDB = persistent NoSQL in RAM (faster, more expensive). |
| **RDS Multi-AZ vs Read Replica** | Multi-AZ = standby copy in another AZ for **failover** (high availability), not for reads. Read Replica = additional read-only copies to **scale read traffic** (and can be cross-region). |

### 6f. Database exam shortcuts

- "Standard SQL database, fully managed" → **RDS**
- "AWS-optimized SQL database, MySQL/Postgres compatible, faster than RDS" → **Aurora**
- "Auto-scaling SQL for unpredictable workloads" → **Aurora Serverless**
- "Petabyte data warehouse for analytics" → **Redshift**
- "Single-digit ms NoSQL key-value, scales infinitely, serverless" → **DynamoDB**
- "MongoDB-compatible managed DB" → **DocumentDB**
- "Cassandra-compatible managed DB" → **Keyspaces**
- "Graph database for relationships" → **Neptune**
- "Immutable ledger / audit trail" → **QLDB**
- "Time-series sensor / metric data" → **Timestream**
- "Speed up reads with an in-memory cache" → **ElastiCache** (Redis or Memcached)
- "Speed up DynamoDB specifically" → **DAX**
- "Migrate a DB to AWS, possibly across engines (Oracle → Aurora)" → **DMS** (+ **SCT** to convert schema)
- "Disaster recovery: my DB needs to fail over to another AZ automatically" → **RDS Multi-AZ**
- "I have too much read traffic on my SQL DB" → **Read Replicas**
- "Need a DB that runs in many regions, active-active, with auto-scaling" → **DynamoDB Global Tables** (or Aurora Global Database for SQL)

## 7. One last mental model: where does each AWS service "live"?

Memorize this hierarchy — it explains a lot of weird exam wording:

```
Globe
 └── Region (e.g. us-east-1) — geographic area, isolated
      └── Availability Zone (AZ) — one or more data centers, isolated power/network (≥3 per Region)
           └── Subnet — a slice of your VPC, lives in exactly one AZ
                └── EC2 instance / ENI — your actual workload
```

- **Global services** (do not pick a region for these): **IAM, Route 53, CloudFront, WAF (for CloudFront), Shield, Organizations**
- **Regional services** (you pick a region): VPC, S3 buckets (yes, S3 is *regional* even though it has a global namespace), EC2, RDS, DynamoDB tables
- **AZ-bound services**: EBS volumes, subnets, EC2 instances themselves

Common exam trick: "I need this to survive the loss of an AZ" → multi-AZ deployment. "Survive the loss of a Region" → multi-Region (CloudFront / Route 53 failover / Aurora Global Database).
