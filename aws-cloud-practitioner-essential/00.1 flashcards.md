## 1. X vs Y Distractor Pairs (the exam's favourite trap)

<details><summary><b>Macie vs GuardDuty vs Inspector</b></summary>

- **Macie** — discovers PII / sensitive data **in S3**
- **GuardDuty** — threat detection from VPC flow logs, DNS logs, CloudTrail
- **Inspector** — vulnerability scanning of **EC2, ECR, Lambda**

Memory: **M**acie = **M**y data (S3). **G**uardDuty = **G**uard the network. **I**nspector = **I**nspect the workload.
</details>

<details><summary><b>Detective vs Security Hub</b></summary>

- **Detective** — investigate root cause of a finding
- **Security Hub** — central dashboard aggregating findings from many security services

Memory: Detective *digs into one case*, Security Hub *displays all cases*.
</details>

<details><summary><b>KMS vs CloudHSM</b></summary>

- **KMS** — managed encryption keys, **shared** multi-tenant HSM, AWS-managed
- **CloudHSM** — **dedicated single-tenant** HSM, FIPS 140-2 Level 3, you manage keys
</details>

<details><summary><b>Secrets Manager vs Systems Manager Parameter Store</b></summary>

- **Secrets Manager** — auto-rotation of secrets, paid, built for DB creds / API keys
- **Parameter Store** — store config + secrets, cheap/free, **no rotation**
</details>

<details><summary><b>WAF vs Shield</b></summary>

- **WAF** — Layer 7 firewall rules: block SQLi, XSS, bad IPs, rate limit
- **Shield** — DDoS protection. Standard = free auto, Advanced = paid + 24/7 DRT
</details>

<details><summary><b>Shield Standard vs Shield Advanced</b></summary>

- **Standard** — free, automatic, layer 3/4 DDoS protection for all AWS customers
- **Advanced** — paid (~$3000/mo), layer 3/4 + **layer 7**, 24/7 DDoS Response Team, cost protection (refund for spike), covers EC2, ELB, CloudFront, Route 53, Global Accelerator
</details>

<details><summary><b>SQS vs SNS vs EventBridge</b></summary>

- **SQS** — message **queue**, point-to-point, **pull** by consumer, durable
- **SNS** — **pub/sub**, push to many subscribers, topic-based
- **EventBridge** — serverless event bus, routes events with **rules**, integrates SaaS sources (Salesforce, Datadog, etc.)
</details>

<details><summary><b>CloudWatch vs CloudTrail vs Config</b></summary>

- **CloudWatch** — metrics, logs, alarms, dashboards (**performance / operations**)
- **CloudTrail** — audit log of every **API call** (**who did what**)
- **Config** — resource configuration history + compliance rules (**what changed, is it compliant**)
</details>

<details><summary><b>Trusted Advisor vs Well-Architected Tool</b></summary>

- **Trusted Advisor** — **automated** real-time checks across 5 categories (cost, performance, security, fault tolerance, service limits)
- **Well-Architected Tool** — **manual** workload review against the 6 pillars
</details>

<details><summary><b>Reserved Instance vs Savings Plan vs Spot</b></summary>

- **RI** — commit to **specific instance type/region** for 1 or 3 yr, up to 72% off
- **Savings Plan** — commit to **$/hr** for 1 or 3 yr, more flexible across instance types/regions. Compute SP > EC2 SP in flexibility
- **Spot** — spare capacity, up to 90% off, can be reclaimed in **2 min** notice
</details>

<details><summary><b>Organizations SCP vs IAM policy</b></summary>

- **SCP** (Service Control Policy) — **account-level guardrail** in Organizations; sets the *maximum* permissions an account/OU can have
- **IAM policy** — identity (user/role/group) permission within an account
- SCP can deny but **does not grant** permissions on its own
</details>

<details><summary><b>Aurora vs RDS</b></summary>

- **Aurora** — AWS-built MySQL/Postgres-compatible engine, **5× faster** than MySQL, cloud-native architecture, auto-scaling storage
- **RDS** — managed standard engines (MySQL, Postgres, MariaDB, Oracle, SQL Server, Aurora)
- Aurora is one of the engines RDS can run, but is positioned as the AWS-native choice
</details>

<details><summary><b>Redshift vs RDS</b></summary>

- **Redshift** — analytics warehouse (**OLAP**), petabyte-scale, columnar, SQL on big historical data
- **RDS** — transactional DB (**OLTP**), row-based, day-to-day app database
</details>

<details><summary><b>CloudFront vs Global Accelerator</b></summary>

- **CloudFront** — global **CDN**, caches content at edge locations
- **Global Accelerator** — static IPs that route users to the **nearest healthy region** over AWS backbone (**no caching**)

Memory: CloudFront = content. Global Accelerator = path.
</details>

<details><summary><b>EFS vs FSx vs EBS</b></summary>

- **EBS** — block storage, attached to **one EC2** (like a disk)
- **EFS** — managed NFS, **multi-EC2** access, **Linux only**
- **FSx** — managed file systems for **Windows / Lustre / NetApp ONTAP / OpenZFS**
</details>

<details><summary><b>MGN vs DMS vs DataSync vs Snow</b></summary>

- **MGN** (Application Migration Service) — **server lift-and-shift** (replaces SMS)
- **DMS** (Database Migration Service) — **database** migration, supports cross-engine (e.g. Oracle → Aurora)
- **DataSync** — fast **online file** transfer to S3 / EFS / FSx
- **Snow Family** — **physical** data transfer (truck/box)
</details>

<details><summary><b>IAM Identity Center vs Cognito</b></summary>

- **IAM Identity Center** (formerly SSO) — sign-in for **employees** to AWS accounts + business apps
- **Cognito** — sign-up/sign-in for **your end-user customers** (mobile/web apps)
</details>

<details><summary><b>Step Functions vs SWF</b></summary>

- **Step Functions** — modern serverless workflow orchestration, state machines (correct answer 95% of the time)
- **SWF** — older workflow service, legacy
</details>

<details><summary><b>Direct Connect vs Site-to-Site VPN</b></summary>

- **Direct Connect** — dedicated **private fiber** from your DC to AWS (no internet), low latency, takes weeks to provision
- **Site-to-Site VPN** — encrypted tunnel **over public internet**, fast to set up
</details>

<details><summary><b>Organizations vs Control Tower</b></summary>

- **Organizations** — multi-account management, consolidated billing, SCPs (the building block)
- **Control Tower** — opinionated multi-account **landing zone** built on top of Organizations (the turn-key product)
</details>

## 3. Cloud Concepts (your weakest domain — 24% of exam)

<details><summary><b>List all 6 Well-Architected pillars</b></summary>

1. Operational Excellence
2. Security
3. Reliability
4. Performance Efficiency
5. Cost Optimization
6. Sustainability

Memory: **OS RPCS** or "**O**ld **S**chool **R**ock **P**aper **C**ost **S**ustain".
</details>

<details><summary><b>List the 6 advantages of cloud (memorize AWS phrasing)</b></summary>

1. Trade capital expense for variable expense
2. Benefit from massive economies of scale
3. Stop guessing capacity
4. Increase speed and agility
5. Stop spending money running and maintaining data centers
6. Go global in minutes
</details>

<details><summary><b>List all 6 CAF (Cloud Adoption Framework) perspectives</b></summary>

1. Business
2. People
3. Governance
4. Platform
5. Security
6. Operations

Memory: "**B**ig **P**eople **G**overn **P**latforms **S**ecurely **O**perating".
</details>

<details><summary><b>Shared Responsibility Model — AWS is responsible for ___, customer is responsible for ___</b></summary>

- **AWS** — security **OF** the cloud: hardware, hypervisor, facilities, managed-service internals (e.g. RDS engine patching, S3 disk replacement)
- **Customer** — security **IN** the cloud: data, IAM users/policies, OS patching on EC2, network config (security groups, NACLs), encryption choices, application code
</details>

<details><summary><b>For RDS, who patches the database engine? Who patches the OS? Who manages backups by default?</b></summary>

- **DB engine patching → AWS** (managed service)
- **OS patching → AWS**
- **Backups → AWS** (automated, 7-day default; you configure window)
- Customer is still responsible for: data, schema, in-DB users, network/IAM access controls, encryption choice
</details>

<details><summary><b>For EC2, who patches the OS?</b></summary>

**Customer.** EC2 is IaaS — AWS gives you the VM, you own everything inside it (OS, middleware, app, data).
</details>

<details><summary><b>IaaS vs PaaS vs SaaS — give an AWS example for each</b></summary>

- **IaaS** — EC2, VPC (raw infra)
- **PaaS** — Elastic Beanstalk, RDS, Lambda (managed platform, you bring code/data)
- **SaaS** — WorkMail, Connect, Chime, Q (finished application)
</details>

<details><summary><b>Three cloud deployment models</b></summary>

1. **Cloud** (public)
2. **Hybrid** (on-prem + cloud)
3. **On-premises** (private)
</details>

<details><summary><b>Region vs Availability Zone vs Edge Location</b></summary>

- **Region** — geographic area (e.g. us-east-1). Has a minimum of **3 AZs**.
- **Availability Zone (AZ)** — one or more discrete data centers within a Region, isolated power/network
- **Edge Location** — CloudFront PoP for caching content close to end users (hundreds globally, far more than Regions)
</details>

<details><summary><b>Minimum number of AZs per AWS Region?</b></summary>

**3**
</details>

<details><summary><b>What is "high availability" achieved by, in AWS terms?</b></summary>

Deploying across **multiple AZs** within a Region. (Multi-Region = disaster recovery, beyond HA.)
</details>

<details><summary><b>"Fault tolerance" — scale up or scale out?</b></summary>

**Scale OUT** (more instances, distributed). Scale up = bigger box = single point of failure.
</details>

<details><summary><b>"Elasticity" vs "Scalability"</b></summary>

- **Scalability** — system can grow to handle more load (often manual / planned)
- **Elasticity** — system **automatically** grows AND shrinks with demand (cloud-specific superpower)
</details>

<details><summary><b>"Agility" in cloud terms</b></summary>

Speed of provisioning and experimenting. Spinning up a server takes minutes, not weeks. Lower cost of failure.
</details>

<details><summary><b>What does "loose coupling" mean and which services enable it?</b></summary>

Components don't directly call each other; they communicate via a **buffer/queue** so one failing doesn't take down the others. Enabled by **SQS** (queue), **SNS** (pub/sub), **EventBridge** (event bus).
</details>

<details><summary><b>Difference between vertical scaling and horizontal scaling, and which is "cloud-preferred"?</b></summary>

- **Vertical** = bigger instance (more CPU/RAM). Has a ceiling, usually requires reboot.
- **Horizontal** = more instances behind a load balancer. **Cloud-preferred**, no ceiling, fault-tolerant.
</details>

<details><summary><b>What is the Well-Architected Tool, and is it free?</b></summary>

A free AWS service that walks you through the **6 pillars** as a questionnaire to assess and improve a workload. Generates an improvement plan and milestones.
</details>

<details><summary><b>"Sustainability" pillar — what does AWS expect you to do for it?</b></summary>

Choose efficient regions, right-size workloads, use managed services / serverless (higher utilization), turn off idle resources, prefer Graviton (ARM) for better perf/watt.
</details>

<details><summary><b>Operational Excellence pillar — one-line summary</b></summary>

Run and monitor systems to deliver business value; **continuously improve** processes and procedures (automation, runbooks, observability, IaC).
</details>

<details><summary><b>Reliability pillar — one-line summary</b></summary>

Recover from failure automatically, test recovery, scale horizontally, stop guessing capacity. (Multi-AZ, backups, auto-scaling, health checks.)
</details>

<details><summary><b>Performance Efficiency pillar — one-line summary</b></summary>

Use the **right resource** for the job; experiment more often; go global in minutes; use serverless where it fits.
</details>

<details><summary><b>Cost Optimization pillar — one-line summary</b></summary>

Stop paying for what you don't use; right-size; use managed services; analyze with Cost Explorer / Trusted Advisor / Compute Optimizer.
</details>

## 4. Pricing models (12% of exam, easy points)

<details><summary><b>On-Demand pricing — when to use, billing increment</b></summary>

- Pay per second (Linux, ≥60s) or per hour (Windows)
- No commitment, no upfront
- Use for short-term, spiky, or unpredictable workloads
</details>

<details><summary><b>Reserved Instance — discount range, commitment, payment options</b></summary>

- Up to **72% off** vs On-Demand
- 1 or 3 year commitment to a **specific instance type, OS, and Region**
- Payment: All Upfront / Partial Upfront / No Upfront (more upfront = more discount)
- Standard RI vs Convertible RI (Convertible = can change instance family, lower discount)
</details>

<details><summary><b>Savings Plans — what do you commit to?</b></summary>

A **dollar-per-hour spend** for 1 or 3 years.
- **Compute Savings Plan** — most flexible (any region, any instance family, EC2/Fargate/Lambda)
- **EC2 Instance Savings Plan** — better discount but locks to instance family + region
</details>

<details><summary><b>Spot Instances — discount, reclaim warning, when to use</b></summary>

- Up to **90% off** vs On-Demand
- AWS reclaims with **2-minute warning** when capacity needed
- Use for fault-tolerant, stateless, batch, CI/CD, big data, ML training
- **Never** use for stateful or production-critical workloads
</details>

<details><summary><b>Dedicated Host vs Dedicated Instance</b></summary>

- **Dedicated Host** — entire physical server reserved for you; **supports BYOL** (Bring Your Own License) for Windows / SQL Server / Oracle; visibility into sockets/cores
- **Dedicated Instance** — runs on hardware dedicated to your account but no socket/core visibility
- Use Dedicated Host when license requires it
</details>

<details><summary><b>What is the AWS Free Tier? Three types?</b></summary>

1. **12-month free** — e.g. 750 hr/mo of t2.micro EC2, 5 GB S3
2. **Always free** — e.g. 1 M Lambda requests/mo, 25 GB DynamoDB
3. **Trials** — short-term (e.g. SageMaker 2 months)
</details>

<details><summary><b>Cost Explorer vs Budgets vs Cost & Usage Report (CUR) vs Pricing Calculator</b></summary>

- **Cost Explorer** — visualize past spend + forecast future (UI)
- **Budgets** — alert when actual or forecast spend exceeds threshold
- **CUR** — most detailed line-item billing data exported to S3 (for BI tools)
- **Pricing Calculator** — estimate cost of a planned (not yet built) architecture
</details>

<details><summary><b>Consolidated Billing — which service enables it and what's the benefit?</b></summary>

**AWS Organizations.** Benefits: one bill across all accounts, **volume discounts pooled** across accounts, free tier shared, and unused RI/Savings Plan capacity shared across accounts in the org.
</details>

<details><summary><b>What does Compute Optimizer do?</b></summary>

ML-powered recommendations to **right-size** EC2, EBS volumes, Lambda functions, and Auto Scaling groups based on actual utilization.
</details>

<details><summary><b>Which pricing dimensions does S3 charge on?</b></summary>

1. Storage (per GB/month, varies by class)
2. Requests (PUT/GET/etc.)
3. Data transfer **out** (data in is free; transfer between S3 and EC2 in same region is free)
4. Management features (Inventory, Analytics, etc.)
</details>

## 5. Security cluster (30% of exam — your second priority)

<details><summary><b>IAM — is it regional or global? Free?</b></summary>

**Global** and **free**.
</details>

<details><summary><b>IAM user vs IAM role — when to use each</b></summary>

- **User** — long-term identity with long-term credentials (password, access keys); for humans or legacy apps
- **Role** — identity that's **assumed** to get **temporary credentials**; for AWS services (e.g. EC2 needs S3 access), federated users, cross-account access
- **Best practice: prefer roles. Never put access keys on EC2 — attach a role.**
</details>

<details><summary><b>What should the root user be used for?</b></summary>

Almost nothing. Only **account-level tasks** that require it (e.g. closing the account, changing support plan, changing root email). Always enable MFA on root.
</details>

<details><summary><b>IAM policy evaluation — explicit deny vs explicit allow vs default</b></summary>

- Default = **deny**
- An **explicit allow** overrides the default deny
- An **explicit deny** overrides any allow (deny always wins)
</details>

<details><summary><b>Security Group vs Network ACL (NACL)</b></summary>

| | Security Group | NACL |
|---|---|---|
| Level | Instance (ENI) | Subnet |
| Stateful? | **Stateful** (return traffic auto-allowed) | **Stateless** (must allow both directions) |
| Rules | Allow only | Allow **and** deny |
| Eval | All rules evaluated | Numbered, lowest first |
</details>

<details><summary><b>3 types of KMS keys</b></summary>

1. **AWS owned keys** — owned and managed by AWS, not in your account, free
2. **AWS managed keys** — created by AWS services on your behalf in your account (e.g. `aws/s3`), free
3. **Customer managed keys** — you create and manage; rotation, policies; **paid** ($1/mo per key)
</details>

<details><summary><b>Which AWS service should I use to grab official compliance reports (SOC, ISO, PCI)?</b></summary>

**AWS Artifact**. Self-service download of AWS's compliance reports and agreements.
</details>

<details><summary><b>Which service centrally manages WAF/Shield/SG rules across multiple accounts?</b></summary>

**AWS Firewall Manager** (requires Organizations).
</details>

<details><summary><b>Network Firewall vs WAF</b></summary>

- **Network Firewall** — managed firewall at the **VPC** level (layers 3-7), inspects/filters/blocks traffic in your VPC
- **WAF** — application-layer (L7) rules attached to CloudFront / ALB / API Gateway / AppSync, focused on web app threats (SQLi, XSS)
</details>

<details><summary><b>Encryption-at-rest defaults: which services encrypt by default vs optional?</b></summary>

- **Encrypted by default**: S3 (since 2023), CloudTrail logs, DynamoDB
- **Optional (opt-in)**: EBS, EFS, RDS, Redshift (you choose at create time)
</details>

<details><summary><b>What does GuardDuty analyze?</b></summary>

Three log sources: **VPC Flow Logs**, **DNS logs**, **CloudTrail event logs**. Uses ML + threat intel to flag suspicious activity. No agents to install.
</details>

<details><summary><b>What does Inspector scan?</b></summary>

**EC2** instances, **ECR** container images, and **Lambda** functions for software vulnerabilities (CVEs) and unintended network exposure.
</details>

<details><summary><b>What does Macie do?</b></summary>

ML-powered discovery of **PII / sensitive data in S3** (credit cards, SSNs, etc.).
</details>

<details><summary><b>Security Hub — what is it for?</b></summary>

**Central dashboard** that aggregates findings from GuardDuty, Inspector, Macie, IAM Access Analyzer, and partner tools. Maps findings to standards (CIS, PCI DSS, AWS Foundational Security Best Practices).
</details>

<details><summary><b>Which security service can investigate the root cause of an alert?</b></summary>

**Amazon Detective.** Builds a graph of behavior to help analyze and visualize security findings.
</details>

## 6. Storage (frequent exam topic, your IT-shaky area)

<details><summary><b>S3 storage classes ordered cheapest → most expensive (by storage cost)</b></summary>

1. Glacier Deep Archive (cheapest, 12 hr retrieval)
2. Glacier Flexible Retrieval (1 min – 12 hr)
3. Glacier Instant Retrieval (ms retrieval, infrequent)
4. One Zone-IA (single AZ, infrequent)
5. Standard-IA (multi-AZ, infrequent)
6. Intelligent-Tiering (auto-moves between classes)
7. Standard (most expensive, instant access)
</details>

<details><summary><b>S3 durability (number of nines)</b></summary>

**11 nines (99.999999999%)** durability across all storage classes.
</details>

<details><summary><b>When to use One Zone-IA?</b></summary>

Infrequently accessed data that **can be recreated** if lost (e.g. secondary backup, derived data). Cheaper because stored in **only one AZ** = lower availability/durability profile but cheaper.
</details>

<details><summary><b>S3 Intelligent-Tiering — what does it do?</b></summary>

Automatically moves objects between access tiers (Frequent / Infrequent / Archive Instant / Archive / Deep Archive) based on access patterns. Small monthly monitoring fee per object. Best when access patterns are unknown or changing.
</details>

<details><summary><b>EBS scope — single AZ or multi-AZ?</b></summary>

**Single AZ.** EBS volume is locked to one AZ. To use in another AZ: snapshot to S3, then create volume in target AZ from snapshot.
</details>

<details><summary><b>EFS — which OS, single or multi-AZ?</b></summary>

**Linux only**, **multi-AZ** by default. Mounted as NFS, can be accessed concurrently by thousands of EC2 instances.
</details>

<details><summary><b>When to use FSx for Windows File Server vs FSx for Lustre vs FSx for NetApp ONTAP vs FSx for OpenZFS</b></summary>

- **Windows** — SMB shares for Windows workloads (Active Directory integrated)
- **Lustre** — high-performance computing, ML training, video processing
- **NetApp ONTAP** — migrating NetApp workloads, multi-protocol (NFS/SMB/iSCSI)
- **OpenZFS** — migrating ZFS workloads, low-latency NFS
</details>

<details><summary><b>Snow Family sizes</b></summary>

- **Snowcone** — handheld, ~8 TB, can be shipped or carried
- **Snowball Edge** — ~80 TB + onboard compute (Storage Optimized or Compute Optimized variants)
- **Snowmobile** — semi-truck, exabytes (being retired but still on exam)
</details>

<details><summary><b>Storage Gateway types</b></summary>

- **File Gateway** — NFS/SMB on-prem appliance backed by **S3**
- **Volume Gateway** — iSCSI block volumes backed by S3 (cached or stored mode)
- **Tape Gateway** — virtual tape library (VTL) backed by S3 / Glacier; replaces physical tape
</details>

<details><summary><b>What service centrally manages backups across many AWS services?</b></summary>

**AWS Backup.** Single console for backup policies across EBS, EFS, RDS, DynamoDB, FSx, Storage Gateway, S3, etc.
</details>

<details><summary><b>S3 versioning + MFA Delete</b></summary>

**Versioning** keeps every version of an object so you can recover from accidental delete/overwrite. Enable at bucket level. **MFA Delete** requires MFA token to permanently delete versions or disable versioning — extra protection on critical buckets.
</details>

## 7. Networking (your IT-shaky area)

<details><summary><b>What is a VPC? Default scope?</b></summary>

A **Virtual Private Cloud** — your isolated private network inside AWS. **Region-scoped** (spans all AZs in that Region). Every account gets a default VPC per Region.
</details>

<details><summary><b>Public vs Private subnet — what's the technical difference?</b></summary>

A **public subnet** has a route in its route table pointing to an **Internet Gateway**. A private subnet does not.
</details>

<details><summary><b>Internet Gateway vs NAT Gateway</b></summary>

- **Internet Gateway (IGW)** — allows resources in public subnets to send/receive traffic to/from the internet (bidirectional)
- **NAT Gateway** — allows resources in **private** subnets to **call out** to the internet (e.g. download patches), but blocks inbound. AWS-managed, lives in a public subnet, charged per hour + per GB
</details>

<details><summary><b>4 ways to connect on-prem to AWS</b></summary>

1. **Site-to-Site VPN** — encrypted tunnel over public internet (fast to set up)
2. **Direct Connect** — dedicated private fiber, low latency, weeks to provision
3. **Client VPN** — individual remote users into a VPC
4. **Transit Gateway** — hub-and-spoke connecting many VPCs + on-prem at scale
</details>

<details><summary><b>What is VPC Peering vs Transit Gateway?</b></summary>

- **VPC Peering** — 1-to-1 connection between two VPCs, non-transitive (A→B, B→C does **not** mean A→C)
- **Transit Gateway** — hub that lets many VPCs and on-prem networks talk to each other through one place, **transitive**, scales much better
</details>

<details><summary><b>What is PrivateLink (VPC Endpoint)?</b></summary>

Privately connect your VPC to AWS services or third-party SaaS **without going over the public internet** or using IGW/NAT. Two flavors: **Interface Endpoint** (most services, ENI-based, paid) and **Gateway Endpoint** (free, only for **S3 and DynamoDB**).
</details>

<details><summary><b>Which two services support a VPC Gateway Endpoint?</b></summary>

**S3** and **DynamoDB** — only these two. All other services use Interface Endpoints.
</details>

<details><summary><b>Route 53 — what does it do beyond DNS?</b></summary>

- DNS hosting
- Domain registration
- **Health checks** with failover routing
- Routing policies: simple, weighted, latency, geolocation, geoproximity, failover, multi-value
</details>

<details><summary><b>CloudFront — what is it and what does it cache from?</b></summary>

Global **CDN** with hundreds of edge locations. Origins can be S3, ALB, EC2, or any HTTP server (even non-AWS). Reduces latency, offloads traffic from origin, integrates with WAF and Shield.
</details>

<details><summary><b>Global Accelerator — when to use vs CloudFront?</b></summary>

Use **Global Accelerator** when:
- You need **static IPs** (e.g. for firewall whitelisting)
- Traffic is **non-HTTP** (TCP/UDP) — e.g. gaming, IoT, voice
- You want fast failover between Regions

Use **CloudFront** when caching cacheable HTTP content close to users.
</details>

<details><summary><b>API Gateway — what is it?</b></summary>

Managed front door for REST / HTTP / WebSocket APIs. Handles auth (Cognito, IAM, Lambda authorizer), throttling, caching, versioning. Often paired with Lambda for serverless APIs.
</details>

<details><summary><b>4 types of Elastic Load Balancer</b></summary>

- **ALB** — HTTP/HTTPS, layer 7, path/host-based routing, websockets
- **NLB** — TCP/UDP/TLS, layer 4, ultra-low latency, millions of req/sec, static IP per AZ
- **GLB** — Gateway Load Balancer; routes traffic through 3rd-party network appliances (firewalls, IDS)
- **CLB** — Classic, legacy, avoid for new
</details>

## 8. Migration (6 R's)

<details><summary><b>List the 6 R's of cloud migration with one example each</b></summary>

1. **Rehost** ("lift and shift") — move VM as-is using **MGN**
2. **Replatform** ("lift, tinker, shift") — small optimizations, e.g. on-prem MySQL → **RDS for MySQL**
3. **Repurchase** — drop it, buy SaaS, e.g. self-hosted CRM → Salesforce
4. **Refactor / Re-architect** — break into microservices, go cloud-native (Lambda, DynamoDB)
5. **Retain** — keep on-prem (regulatory, recently invested, etc.)
6. **Retire** — turn it off, no one uses it
</details>

<details><summary><b>Which service replaces Server Migration Service (SMS) for lift-and-shift?</b></summary>

**Application Migration Service (MGN)**.
</details>

<details><summary><b>Which service migrates databases including across engines (e.g. Oracle → Aurora)?</b></summary>

**DMS** (Database Migration Service). Use **Schema Conversion Tool (SCT)** to convert schema before/during DMS migration when changing engines.
</details>

<details><summary><b>Which service gives a single dashboard tracking all migrations?</b></summary>

**Migration Hub.**
</details>

<details><summary><b>Which service discovers on-prem servers and apps before migrating?</b></summary>

**Application Discovery Service.** Agentless connector or agent-based; feeds Migration Hub.
</details>
