# IAM principal

## what is a principal?

a **principal** is whoever (or whatever) is making a request to AWS. it is the entity AWS authenticates and then checks permissions for.

every AWS API call answers two questions:
1. **who is making this request?** → the principal
2. **are they allowed to do it?** → evaluated against IAM policies

## what can be a principal?

| principal type | example | when it shows up |
|---|---|---|
| AWS account (root user) | `arn:aws:iam::123456789012:root` | account owner; should rarely be used |
| IAM user | `arn:aws:iam::123456789012:user/alice` | a human or app with long-lived credentials |
| IAM role (when assumed) | `arn:aws:iam::123456789012:role/EC2-S3-Role` | EC2 / Lambda / another service assumes it |
| federated user | SAML / OIDC identity from corp IdP, Google, etc. | SSO logins, web identity federation |
| AWS service | `lambda.amazonaws.com`, `ec2.amazonaws.com` | when one AWS service acts on your behalf |
| anonymous / public | `"*"` | public access (e.g. an open S3 bucket) |

a principal is not always a person. it is often a piece of software, an EC2 instance, or another AWS service.

## "Principal" inside a policy document

the literal `Principal` field appears in **resource-based policies** — policies attached to a *resource* rather than to an identity.

example: an S3 bucket policy that lets a specific IAM user read objects.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:user/alice"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

read in english: "the resource (this bucket) **allows** the principal (`alice`) to perform `s3:GetObject` on `my-bucket/*`."

the `Principal` element answers: **"who is this rule about?"**

## identity-based vs resource-based policies

| | identity-based policy | resource-based policy |
|---|---|---|
| attached to | an IAM user, group, or role | a resource (S3 bucket, SQS queue, KMS key, Lambda, SNS topic, etc.) |
| says | "this identity can do X on resource Y" | "these principals can do X on me" |
| has a `Principal` field? | **no** — principal is implicit (whoever the policy is attached to) | **yes** — must explicitly name who is allowed |
| examples | IAM policies | S3 bucket policies, SQS queue policies, KMS key policies, Lambda resource policies, SNS topic policies |

so:
- an IAM policy attached to user Alice → no `Principal` field, because it *is* about Alice.
- an S3 bucket policy → must specify `Principal`, because it needs to declare which outside identities it applies to.

## common forms of `Principal`

specific IAM user
```json
"Principal": { "AWS": "arn:aws:iam::123456789012:user/alice" }
```

specific IAM role
```json
"Principal": { "AWS": "arn:aws:iam::123456789012:role/MyRole" }
```

an entire AWS account (any IAM identity in that account, subject to its own IAM policies)
```json
"Principal": { "AWS": "arn:aws:iam::123456789012:root" }
```

an AWS service (typical in trust policies for IAM roles)
```json
"Principal": { "Service": "lambda.amazonaws.com" }
```

a federated identity provider
```json
"Principal": { "Federated": "accounts.google.com" }
```

everyone (public) — use with extreme care
```json
"Principal": "*"
```

## where it matters: trust policies on IAM roles

every IAM role has a **trust policy** (a resource-based policy on the role itself). it uses `Principal` to say **who is allowed to assume this role**.

example — a role that an EC2 instance can assume:

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Service": "ec2.amazonaws.com" },
    "Action": "sts:AssumeRole"
  }]
}
```

read: "allow the EC2 service to assume me." that is why the principal here is a service, not a user.

## quick mental model

- **principal = the "who"** in any AWS authorization decision.
- in **identity-based policies**: the principal is implicit (the identity the policy is attached to).
- in **resource-based policies**: the principal is explicit, named in the `Principal` field.
- a principal can be: a user, role, account, service, federated identity, or `*` (anyone).

## exam tips

- `Principal` literally appears only in **resource-based policies** and **role trust policies**.
- IAM policies attached to users / groups / roles **do not have** a `Principal` field — a giveaway in policy-reading questions.
- for **cross-account access**: typically a resource-based policy names the other account or role as `Principal`, **plus** an identity-based policy in the other account allows its identity to make the call. **both sides must allow it.**
- `Principal: "*"` with no `Condition` = public access — a major security finding.
